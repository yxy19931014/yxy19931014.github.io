Veröffentlicht am 2016-02-05

node

node相关概念(node 入门)

  1. node是一个构建于chrome v8引擎之上的一个JavaScript运行时环境
  2. node可以像浏览器一样去解析和执行JavaScript代码
  3. JavaScript是一个动态脚本语言
  • 预编译
  • 再执行
  1. JavaScript之前的解析和执行一直停留在浏览器环境
  • 通过编译解析引擎:解析和执行JavaScript脚本代码
  • chrome v8 引擎是所有浏览器里面解析和执行js代码最快的
  • node作者把chrome 的v8引擎移植出来,又基于v8 引擎之上构建了一系列的API,在node中没有界面,不再需要关心HTML,css,只需关注JavaScript, 和dom ,style 等没有任何关系
  • node的操作:文件操作,网络操作,让JavaScript具有了服务器编程能力
  1. node的运行机制
  • 事件驱动
  • 非阻塞IO模型: 大量的异步编程接口( 充满了类似于客户端ajax一样的 api, 异步和回调紧紧相连)
  • 跨平台
  • 单线程’
  • 轻量和高效
  1. npm
  • npm是世界上最大的开源库生态系统
  • 在npm上发布的第三方包已经接近35 万个之多
  • 包只能解决某一个单一的问题,但解决不了业务

    node 应用

  1. 网站开发
  2. 数据库交互
  3. 移动应用
  4. 网络游戏
  5. cli (Command line Interface)程序:
  6. node具有开发命令行工具的的能力
  • webpack 打包构建工具
  • http-server 在终端快速提供一个服务
  • browser-sync 提供服务,提供浏览器自动刷新的能力

    node 局限

  • 对io 操作比较频繁的操作,
  • 文件读写
  • 和数据库交互,网络数据io
  • 实时通信
  • 聊天,微信,qq,直播

        path 环境变量

    1. 问题: 为什么在终端的任何目录下输入’node’命令,执行的都是’node.exe’这个可执行文件
  • 当在终端输入一个字符串的时候,终端会在当前目录下找和该字符串相关的执行文件,如果找不到优先进入用户path 环境变量
  • path 环境变量通过英文分号作为分割保存多个目录
  • 终端会一个一个的进入该目录查找该字符串对应的文件
  • 如果找到就会执行
  • 找不到就会报错
    1. 把可执行文件的所属目录的绝对路径放到path中

      node中的JavaScript

      EcmaScript

  • var , if , function, {}, object,Number,String,Date,Math.. console, 定时器函数,global 全局

    核心模块

  • 核心模块是由node 环境平台本身提供的,以一个唯一的核心模块标识名进行暴露,使用方法的时候直接通过require 函数进行加载
  • node 把绝大多数的api都封装到一个具名的核心模块中,例如文件操作的’fs’,网络操作的’http’
  • 也就是想使用核心模块,必须加载才可以使用

    第三方模块

  • 由社区提供(公司和个人)
  • 例如: moment :时间处理类库
  • 如何在node中引入第三方包,
    1. 下载
    2. 通过require函数引包
    3. 查看moment api文档
      ###用户自定义模块
  • 在node中,也提供了一种所谓JavaScript模块化解决方案,(类似于之前学过的requirejs)
  • require是一个函数,可以在node中的任何脚本文件中使用,用来加载模块(加载解析执行)
    1. 解析执行被加载文件中的代码
    2. 在node中,一个文件就是一个模块
    3. 模块天生就是一个私有的作用域
    4. 每个模块提供了一个接口对象:module,通过module.exports导出内部变量; 别的模块通过require加载会得到指定模块,得到该模块内部的变量

      异步

  • readFile
  • 通过一个回调函数来获取异步的操作结果
  • 如果操作成功,err是null,失败就是错误对象
  • 所以在异步操作中优先判断一下err的值
  • 异步操作中无法通过try-catch来捕获异常
    ###同步
  • readFileSync
  • 和普通代码一样,像接收变量返回值一样接收结果
  • 可以用try-catch捕获异常

    关于文件操作中同步和异步

  • 优先使用异步

    web 路径

  • 在web开发中,看到的路径不一定是真实的文件路径
  • 看到的文件名也不一定是真实的文件名
  • URL本质上只是一个标识而已
  • 绝大多数情况下,需要开发人员自定义标识
    1. 为了好看,路由更简洁
    2. 为了更加灵活的控制
  • 网站中的静态资源以/开头,避免了URL受当前文件路径的影响

    node 中的相对路径

  • node中,文件操作中的相对路径永远都是相对于执行node命令所处的终端路径
  • 解决方法: 把相对路径转换为绝对动态路径
    ###在每个模块中都提供给了两个属性(动态绝对路径)
  • __dirname 可以动态的获取当前文件模块的绝对路径,利用这个属性可以动态的拼接一个动态的绝对路径
  • __ filename
  • 虽然文件的位置可能变化,但是这两个属性也会随之变化
  • 所以就可以得到文件的动态绝对路径

    node中的path 模块

  • path.join()方法可以拼接路径(其中../可以自动回到上一级目录)

    操作文件的相对路径的应用场景

  • 一般操作文件的相对路径只有在开发一些全局命令行工具的时候会用到.

    post请求

    没有文件上传的post表单提交

  1. 监听req事件和end事件
  2. 如果表单数据比较大,则 data 事件可能会触发多次，每次触发执行都可以通过回调函数拿到一个数据块, 然后在外部定一个变量，每接收到一个数据块，则将该数据块与外部变量拼接起来,最后在 end 事件中就可以使用收集完的完整数据块了

    node中的模块化

  • 规范:Common JS
  • node中的模块化采用了CommonJS 规范,CommonJs不仅是模块规范,他为JavaScript提供了
  • module.exports和exports的原本指向是相同的,但是任意改变其中一个的指向,则两者就不再相同; 模块中导出的部分是module.exports,而不是exports.
    ###模块加载的规则
  1. 一个模块被加载过后,就会缓存起来,下次使用的时候直接从缓存中拿,所以不会加载两次
  2. 加载的是模块中的module.exports 的值,而不是引用
  3. 加载模块是require函数接收的参数叫模块标识符
  4. 如果是加载核心模块,先看缓存,如果有,直接拿,没有就直接加载
  5. 自己写的模块,以’./‘ ,’../‘ ,’/‘ 开头(/ 是当前文件模块所属磁盘根目录
  6. 第三方模块包
  7. 先基于当前文件模块所属目录找node_modules目录
  8. 如果找到,则去该目录中着对应的目录
  9. 如果找到,则找该目录中的package.json 文件
  10. 如果找到package.json文件,则找该文件中的main属性
  11. 如果找到main属性,则拿到该属性对应的文件路径
  12. 如果找到moment 目录后,发现没有package.json,或者有package.json没有main属性,或者有main属性但是指向是的路径不存在;则node会默认去看moment 目录下有没有index.js,node-index.json文件; 如果找不到index或者找不到moment或者 找不到 node_modules,则进上一级目录找node_modules(查找规则同上) ,如果上一级还找不到,继续向上,一直到当前文件所属磁盘根目录. 如果最后到磁盘根目录还找不到,最后报错

    umd 规范 兼容了多种JavaScript模块规范

  • 这种模块一般和具体的环境无关

  3. XMLRequestHttp2 提供了formData 可以上传二进制数据,也就是可以上传文件

url 核心模块

  1. 引入
  2. var parseObj=url.parse(req,url,true); 得到请求信息对象
  3. parseObj中有pathname,是请求路径
  4. parseObj 中的query 中是请求参数对象

    MVC 模式,mvvm

  • MVC是一种软件开发模块化架构方式,一种非常好的代码架构组织方式,(适用于web开发),被很多领域应用了,就是用来解耦视图和数据,利用c控制器去调度视图和模型数据的交互
  • 优点:
    • 提高了代码的可维护性
    • 视图数据和业务数据是完全分开的
    • 例如表单提交上来的数据就是视图数据
    • 业务数据就是真实的数据库中的数据
    • 增加了代码的可重用性

      mvvm 的核心就是数据驱动视图

  • 各部分各司其职
  • model不会直接操作数据
  • view也不会直接操作数据
  • 之间通过控制器驱动

    图片上传预览

  • 当用户选择了图片，监听file控件的change事件
  • 在 change事件中将用户选择的文件异步提交到服务器
  • 服务器将图片保存，响应回来该图片的请求地址
  • 将预览图的src设置为服务器的响应资源的路径

    渲染方法

  • 如果要处理SEO，一定要服务端渲染
  • 如果不考虑，则优先客户端渲染，
  • 客户端渲染的好处：减轻服务端压力，提高用户体验
